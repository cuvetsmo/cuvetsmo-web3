# Bug hunt 2026-05-27 · BE-fix postmortem + sweep

After the gasless VetSBTCard claim failed with `InvalidYear()` (selector `0x87870bd7`) the team ran a system-wide bug hunt for similar pattern classes. This doc records what was found, what was fixed, what's documented-and-deferred, and the learning principles to bake into future code reviews.

---

## The trigger bug — recap

```
sender:           0x1848…e388   ← Smart Account derived from a Privy embedded wallet
contract:         0xE679d2…E6F5 (VetSBTCard)
callData function: claim(uint16=2023, ...)
contract guard:   if (yearAdmitted < 2500 || yearAdmitted > 2700) revert InvalidYear();
```

**Root cause**: the contract enforces Buddhist Era (พ.ศ., 2500–2700) but the React form defaulted `String(CURRENT_YEAR - 3)` = `2023` (Gregorian). The bug had been live since the component first shipped — no one had successfully claimed before.

**Direct fix**: defaulted to `CURRENT_YEAR_BE - 3` (= `2566` for Vet 86) + clarified the label as `"ปีที่เข้า / Year admitted (พ.ศ.)"`. Affects both gasless + EOA claim paths. Shipped in commit `3a2975c`.

## Pattern classes swept

### 1 · Unit / format / era mismatches between frontend and contract

| Place | Status | Notes |
|---|---|---|
| `vet-card.tsx` yearAdmitted | ✅ fixed | BE not AD |
| `lib/eas.ts` SCHEMAS.VET_CARD | ✅ uint16 passthrough (schema doesn't impose unit semantics) | If anyone ever wires an EAS attestation flow for VetCard, send BE not AD |
| `subgraph/src/mappings.ts` | ✅ stores raw uint16 from event (whatever the contract wrote) | Indexer is unit-agnostic, ok |
| Other date/time fields | ✅ all use `Math.floor(Date.now() / 1000)` (seconds-since-epoch · same as solidity `block.timestamp`) | No drift |

**Principle**: every numeric value crossing the frontend→contract boundary needs an explicit unit declaration somewhere — in the contract NatSpec, the schema doc, or the env-var comment. Solidity types (uint16) tell you the byte width but never the semantic unit. Make the assumption visible in code; reviewers can then challenge it.

### 2 · Smart Account vs EOA read-write divergence (post C2 gasless rollout)

This is the **bigger latent class** uncovered by the bug hunt. C2 introduced a second on-chain identity per user: the Coinbase Smart Account derived from the Privy EOA. Reads that target only the EOA miss everything minted via gasless flows.

| Place | Bug | Fix |
|---|---|---|
| `vet-card.tsx` `VetCardPanel` `cardOf(EOA)` only | 🔴 user who claimed gasless sees ClaimCardView forever · tries again → `AlreadyClaimed` revert | ✅ fixed · added second `useReadContract` for Smart Account address · `hasCard` is OR of both · ShowCardView gets `heldBySmartAccount` flag + visible badge |
| `profile-panel.tsx` `cardOf(EOA)` + `badgesOf(EOA)` | 🔴 gasless claims invisible · Smart-Account-held badges hidden | ✅ fixed · dual reads + merged `ownedBadgeIds` set |
| `app/api/quests/[id]/verify/route.ts` `receipt.from !== userAddress` (line 128-129) | 🟡 latent · breaks if ANY future quest is completed via a gasless ERC-4337 UserOp (receipt.from = bundler, not user) | ⏳ deferred · no gasless quest wired today · documented below |

**The dual-read helper** lives at `lib/use-user-addresses.ts` — single `useUserAddresses()` hook returning `{ eoa, smartAccount, ready, smartAccountResolved }`. Wire it into any component that reads user-state from a contract.

## Latent issues documented (not fixed this session)

### L1 · Quest verify breaks for gasless quest tx (when /play/mint or similar goes gasless)

`/api/quests/[id]/verify` line 128–135 checks `receipt.from === userAddress`. For a Pimlico-sponsored UserOp, `receipt.from` is the bundler/EntryPoint, not the user's Smart Account. So any future gasless mint/transfer/approve quest will fail this check.

**Fix when needed**: parse `UserOperationEvent(bytes32 userOpHash, address indexed sender, ...)` log from the receipt and check `sender` against the Smart Account address (or against `userAddress` if frontend starts sending Smart Account address as userAddress for gasless paths).

### L2 · Profile badge listing reads BadgeRegistry · attestations now go to EAS

Quest verify route was refactored to issue EAS BADGE attestations (B2) instead of BadgeRegistry mints. `profile-panel.tsx` still reads `badgesOf` from BadgeRegistry → returns `[]` for everyone going forward.

**Today's stopgap**: localStorage `completed.ids` is the user-visible source of truth. Works for self-state but not portable to other wallets / not on-chain visible.

**Fix when needed**: read EAS BADGE attestations by recipient (use `getAttestationsByRecipient(addr, { schemaUid: SCHEMAS.BADGE.uid })` already in `lib/eas.ts`) and render them as badge entries. Goldsky subgraph would speed this up — see `docs/INDEXER_SETUP.md`.

### L3 · ShowCardView SVG uses Smart Account address when held there

The SVG card renders the wallet address `WALLET 0x96…c456` (Smart Account) instead of the user's normal EOA. Currently OK — we surface a "🎁 Smart Wallet (gasless)" badge above so users understand. Long-term may want to render BOTH addresses on the card or just the EOA + an attestation badge.

### L4 · Guestbook entries show Smart Account sender for gasless posts

Same shape as L3 · ok for Phase 0 (educational visibility into how AA works) · document in /play/board copy if it confuses users.

## Principles to bake into reviews

1. **Every wagmi `useReadContract` that targets a user-state mapping** (cardOf, badgesOf, balanceOf, etc.) **must accept both the EOA and the Smart Account address** going forward. Use `useUserAddresses()` and read at both. Merge results.

2. **Every msg.sender comparison server-side** (verify routes, off-chain attesters) **must account for ERC-4337 indirection**. Either parse UserOperationEvent or accept multiple expected sender addresses.

3. **Every numeric arg crossing the FE→contract boundary needs a unit comment** (BE vs AD, seconds vs ms, wei vs gwei, etc.). The Solidity type tells you nothing about semantics.

4. **Every backend mint/attest target must match the frontend read source**. If verify route issues EAS attestations, profile must read EAS. If verify route mints BadgeRegistry, profile must read BadgeRegistry. Drift between these = phantom-data UX bugs.

5. **Every env-var read must have a graceful fallback** OR explicit `throw new Error("X missing")` at boot. Never silent-undefined.

6. **Iron Rule 0 for status reports**: when the user reports a transaction reverted with a 4-byte selector, decode it (compare with the contract's custom-error list) BEFORE guessing. The selector `0x87870bd7` was unique to `InvalidYear()` — diagnosis takes 30 seconds.
